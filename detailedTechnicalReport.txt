Phase 1: Core Architecture & The Real-Time Engine
In this phase, the objective was to build a system capable of handling 50+ concurrent connections, managing live state across all of them without a dedicated backend server, and ensuring players could securely log in and see their team data.

1. The Tech Stack Selection
In an interview, if they ask why you didn't use React or Node.js, your answer is about efficiency and latency.

Frontend: Pure Vanilla HTML5, CSS3, and modern JavaScript (ES6 Modules). By skipping heavy frontend frameworks, you eliminated build times and client-side rendering overhead, keeping the app lightning-fast even on weak mobile connections.

Backend: Google Firebase Firestore. This is a BaaS (Backend-as-a-Service). You chose Firestore specifically because it uses WebSockets to push data to the client instantly, rather than forcing the client to constantly ask the server, "Are there new points?" (HTTP Polling).

Environment: Windows Subsystem for Linux (WSL2) with VS Code Live Server.

2. ES6 Module Architecture (Separation of Concerns)
Instead of writing one massive, unreadable JavaScript file, the project was architected using ES6 Modules (type="module" in the HTML).

config.js: Acts as the single source of truth for routing games.

auth.js: Handles session state and UI gating.

db.js: The exclusive bridge to Firebase. No other file is allowed to talk to the database directly.

main.js: The frontend controller that updates the DOM (Document Object Model) based on database triggers.

Interview flex: This structure demonstrates the Single Responsibility Principle. If the database breaks, you only check db.js.

3. The Authentication Gateway (auth.js)
You needed a way to identify players without forcing them to create actual accounts with passwords.

Implementation: A pseudo-auth system utilizing the browser's localStorage.

Logic: When a user selects their Triad and enters an alias, the script packages that into a JSON object and saves it to the local browser memory.

Benefit: If a user accidentally refreshes the page or closes the tab on their phone, auth.js intercepts the load, checks localStorage, and instantly bypasses the login gate, keeping them in the game seamlessly.

4. The Real-Time Event Listener (db.js)
This is the heart of the application. You utilized Firestore's onSnapshot method.
Instead of writing a function to "fetch" the leaderboard, onSnapshot opens a persistent, two-way WebSocket connection. The moment the Admin clicks "Deploy," or points are injected into the cloud, Firebase blasts that update down the socket, and your main.js script dynamically rebuilds the HTML leaderboard in milliseconds.

The Roadblocks & Engineering Solutions (Interview Goldmine)
Interviewers love hearing about things breaking. Here are the two major technical hurdles from Phase 1 and how they were solved.

Critical Failure 1: The ES6 Strict Mode Crash

The Bug: The entire site silently failed to load its logic. Buttons wouldn't click, and no errors popped up in the UI.

The Cause: You placed a debugging console.log at the very top of db.js. Because the script was defined as a type="module", the JavaScript engine runs in "Strict Mode." Strict Mode mandates that all import statements must be the absolute first execution lines in the file.

The Fix: Moving the imports to line 1. It taught you how unforgiving modern JS compilation can be compared to older script tags.

Critical Failure 2: The WSL2 WebSocket Blockade

The Bug: Firebase absolutely refused to connect. It wouldn't read or write data, but it didn't throw a standard offline error either.

The Cause: You were running your local server through the Linux Subsystem (WSL2) on a Windows machine. WSL2 has a highly restrictive virtual network bridge. Firestore attempts to open a persistent TCP WebSocket for real-time data. The WSL2 firewall silently swallowed these WebSocket packets, causing the app to hang infinitely while waiting for a handshake.

The Solution: You engineered a bypass by injecting { experimentalForceLongPolling: true } into the Firebase initialization.

Why it worked: Long Polling forces Firebase to abandon the blocked WebSocket protocol. Instead, it holds open a standard HTTP request for as long as possible until the server replies. HTTP traffic easily bypasses the WSL2 virtual network restrictions.
Phase 2: The Iframe Engine, Degrading Timers, and the State Machine
In this phase, the objective was to dynamically load different games into the dashboard without ever refreshing the page. If the page refreshes, the WebSocket connection to Firebase drops and has to be re-established, which causes lag.

1. The Iframe Sandbox Architecture
To keep the main dashboard static while the games rotated, you built an Iframe Engine.

The Logic: The index.html holds the header, the leaderboard, and an empty <iframe id="game-frame">. When the Admin deploys a round, config.js routes the specific game file (like games/round1/quiz.html) directly into that iframe.

The SDE Flex: This is a perfect example of Component Isolation. If Round 2's code completely crashes, it only crashes inside the iframe. The main dashboard and the global leaderboard listener remain 100% operational.

2. Cross-Window Communication (The postMessage API)
Because the game runs inside an iframe, it is functionally a separate webpage. It doesn't have direct access to the updatePlayerScore function sitting in the parent window's main.js.

The Solution: You implemented the window.parent.postMessage() API.

How it works: When a player gets a correct answer in Round 1, the iframe packages a data object { type: 'PHANTOM_SCORE_UPDATE', score: 8 } and fires it up to the parent window. The parent window has a dedicated addEventListener('message') that catches this payload and securely writes the score to Firebase.

3. Round 1: The Degrading Timer Engine
Round 1 wasn't a standard quiz; it featured a dynamic, degrading score mechanic.

The Logic: You initialized a setInterval loop running every 100 milliseconds (0.1s).

The Math: currentPoints = Math.max(0, Math.ceil(timeRemaining));

Why this is smart: By using Math.ceil, you didn't have to write complex if/else statements for the score. At 9.8 seconds, Math.ceil(9.8) automatically equals 10 points. At 8.1 seconds, it equals 9 points. This kept the browser's CPU usage incredibly low during rapid DOM updates.

4. The Admin State Machine (Lock/Unlock Protocol)
You needed absolute control over when 50+ participants started playing to ensure fairness.

The Logic: You created a gameState document in the admin Firebase collection containing a gameActive: false boolean.

The Execution: Inside the iframe, you initialized a secondary, lightweight Firebase listener. The "Initiate Sequence" button starts fully disabled. When you click "Unlock Play Button" on the Admin Dashboard, Firebase flips that boolean to true, instantly broadcasting to all 50 laptops and illuminating the button in neon green.

The Roadblocks & Engineering Solutions
Here are the technical hurdles from Phase 2 to highlight in an interview:

Critical Failure 1: The HTML Parsing Sabotage

The Bug: During the initial setup of the iframe rendering, the layout completely broke.

The Cause: A copy-paste error caused the </body> and </html> closing tags to be injected directly inside the <head> tag of index.html. Browsers try to "auto-fix" broken HTML, which resulted in the browser prematurely closing the document structure and rendering the iframe outside of the intended CSS containers.

The Solution: A thorough DOM (Document Object Model) audit. You had to manually trace the node tree, strip the corrupted tags from the head, and properly close the document. This proves your understanding of how the browser's rendering engine actually parses HTML.

Critical Failure 2: Silent Button Failures in the Admin Panel

The Bug: When clicking the "Deploy" or "Unlock" buttons on the Admin panel, nothing would happen. No round would deploy, and no error appeared on the screen.

The Cause: Network latency or temporary Firebase disconnects were causing the asynchronous await setDoc() functions to hang. Because there was no error handling, the script just died silently.

The Solution: You wrapped every Admin function in try...catch blocks and wired the catch directly to the Admin UI. If Firebase failed to write the state, the UI immediately displayed [ERROR]: DEPLOY FAILED in red.

The Takeaway: This shows an interviewer that you build "defensive code." You don't just assume the database will always work; you build UI fallbacks for when it inevitably fails.
Phase 3: The "Phantom Player" Injection & Master Clock Synchronization
The objective here was twofold: figure out a way to seamlessly merge manual judge scores into an automated leaderboard (Round 2), and completely synchronize the UI of 50 different laptops so no team got an unfair time advantage (Round 3).

1. The "Phantom Player" Data Architecture (Round 2)
You had a live leaderboard dynamically calculating totals based on player aliases within a Triad Unit. But for the Scavenger Hunt, the judges needed to inject bulk points directly to a team, not a specific player.

The Problem: Modifying the main.js leaderboard listener to look for a separate "bonus points" database collection would require rewriting the entire frontend logic and adding unnecessary read operations, costing performance.

The Solution: The "Phantom Player" bypass. You engineered addTeamBonus() to create a dummy document in the players collection named JUDGE_OVERRIDE.

The SDE Flex: The frontend listener in main.js didn't need a single line of code changed. It automatically swept up the JUDGE_OVERRIDE points and aggregated them into the Triad's total score seamlessly. You solved a complex UI problem entirely through clever database structuring.

2. Atomic Operations and the increment() Method
When the Admin slams the "Inject Points" button, you didn't use a standard read/write cycle.

The Implementation: You explicitly imported Firebase's increment() function for the hunt_points field.

Why this matters (Interview Gold): If you use traditional logic (Read current score -> Add 50 -> Write new score), you risk a Race Condition. If two judges hit inject at the exact same millisecond, they both read "100", add 50, and write "150", effectively deleting 50 points. increment() is an Atomic Operation. It tells the server directly, "Whatever this number currently is, add 50 to it," entirely eliminating network race conditions.

3. The Master Clock Synchronization (Round 3)
Round 3 (Witty Captions) required 15 seconds of typing followed by 15 seconds of voting.

The Problem: Relying on setTimeout or setInterval on 50 local browsers is disastrous. If a team has high latency and logs in 4 seconds late, their local 15-second timer will run long, allowing them to type while everyone else is already voting.

The Solution: You stripped the timing authority away from the clients and built a Master State Machine in the Admin dashboard. The Admin pushes a state (r3Phase: 'input', 'vote', or 'wait') to Firebase.

The Result: The participant screens act purely as "dumb terminals." They listen to the gameState document. When the Admin clicks the button, the Firebase WebSocket blasts the state change to all 50 laptops simultaneously, swapping their DOM views (classList.add('active')) at the exact same millisecond, ensuring absolute competitive fairness.

4. Algorithmic Auto-Scoring & The Self-Voting Block
To prevent chaos during the voting phase, you implemented strict client-side and server-side logic.

Anti-Cheat Logic: When mapping the database snapshot to the voting grid, you added the if(d.triad !== user.triad) logic. This prevented the DOM from even rendering the user's own caption, hard-blocking teams from spamming votes for themselves.

The Auto-Award Algorithm: Instead of manually counting votes, you built the awardR3Winners() function. With a single click, it queried all captions, ran a JavaScript .sort((a, b) => b.votes - a.votes) array method to order them highest-to-lowest, sliced the top 3, and looped them through the addTeamBonus() Phantom Player function to automatically award the 10, 7, and 5 points.

The Roadblocks & Engineering Solutions
Critical Failure 1: Cross-Platform File Pollution

The Bug: While loading the image assets for Round 3, the asset folder became cluttered with broken, unreadable files.

The Cause: You discovered the "Mark of the Web" stream (.Zone.Identifier files). Because you were downloading assets via Windows but hosting them in a Linux filesystem (WSL2), Linux didn't understand the hidden Windows security metadata and split it into literal ghost files.

The Solution: Using bash terminal commands (rm *Zone.Identifier*) to batch-purge the metadata. This proves you have fundamental command-line interface (CLI) and Linux filesystem proficiency, which is crucial for modern DevOps and SDE roles.

Critical Failure 2: Accidental Double-Injections

The Bug: During manual testing of the Admin override, it was incredibly easy to accidentally click the "Inject" button twice, giving a team double points.

The Solution: UI State resetting. You modified the event listener so that the exact millisecond the async injection finishes, the inputManualPoints.value = '' command fires. By forcefully wiping the input field, the Admin cannot double-click the button, as the validation block (isNaN(points)) will instantly catch the empty field and block the second submission.

This phase proves you understand how to protect data integrity and synchronize heavy multiplayer traffic.

Phase 4: Static State Management, CI/CD Pipelines, and Production Readiness
The objective for this final phase was to deploy the frontend to a global edge network, automate future updates, and securely sanitize the database so it was completely pristine before 50+ real users hit the servers.

1. Static State Management (Round 4: Shark Tank)
After building highly complex data flows for Rounds 1-3, Round 4 required almost zero database interaction.

The Engineering Decision: Instead of over-engineering a complex presentation queue in the database, you recognized that Round 4 was a physical/verbal presentation. The app simply needed to act as a synchronized digital billboard.

The Implementation: You reused the gameState document listener. When the Admin changes currentRound to 4 and unlocks it, the app dynamically alters the CSS classes to trigger a pulsating neon green animation (@keyframes pulse).

The SDE Flex: Knowing when not to use the database is just as important as knowing how to use it. By keeping Round 4 static and relying entirely on CSS animations triggered by a single boolean flip, you saved bandwidth, reduced potential points of failure, and minimized server read/write costs.

2. The Vercel CI/CD Pipeline (Continuous Integration / Continuous Deployment)
Deploying the application manually via FTP or zip files is an outdated practice. You implemented a modern CI/CD pipeline using GitHub and Vercel.

The Architecture: You linked your GitHub repository directly to Vercel's edge network.

How it works: Vercel installed a Webhook on your GitHub repository. Now, whenever your teammate finishes their code, pushes a branch, and you merge their Pull Request into the main branch, that Webhook fires. Vercel automatically detects the new code, pulls it down, builds it, and deploys it to the global Content Delivery Network (CDN) in seconds.

Why this impresses interviewers: It proves you understand the modern DevOps lifecycle. You didn't just write code; you built an infrastructure where code automatically ships itself to production the moment it passes review.

3. Database Sanitization (The Production Wipe)
Before the live event, the Firebase database was filled with dummy data, phantom test players, and fake captions.

The Design Choice: You actively chose not to build a "Reset Database" button into the Admin UI.

Why? (The Principle of Least Privilege): If a "Nuclear Reset" button existed on the dashboard, an organizer could accidentally click it during the live event, instantly deleting the scores of all 50 laptops with zero chance of recovery.

The Execution: You separated the environments. The Admin UI is strictly for running the game. Database sanitization (wiping the players and r3_captions collections) was performed manually through the Firebase Console. This hard-separated administrative control from developer-level infrastructure control, ensuring absolute data safety during the live run.

The Roadblocks & Engineering Solutions
Here are the technical insights from this phase to highlight in an interview:

Engineering Concept 1: Avoiding "State Drift" During Deployment

The Scenario: When you deploy updates to a live website, users who already have the site open in their browsers might be running old JavaScript code while the server expects new data structures.

The Solution: Because your architecture was serverless and relied on Firebase WebSockets rather than strict backend APIs, your platform was highly resilient to state drift. The HTML/CSS could be updated on Vercel without breaking the ongoing Firebase WebSocket connections in the participants' browsers.

Engineering Concept 2: The WSL-to-Cloud Authentication Gap

The Scenario: Pushing code from a Linux Subsystem (WSL2) to GitHub for the first time resulted in strict authentication rejections (fatal: Authentication failed).

The Cause: GitHub deprecated standard password authentication for terminal operations to prevent brute-force attacks, requiring token-based access. Furthermore, WSL terminals hide input streams during password pasting, leading to user confusion and duplicate command errors (the command not found loop).

The Solution: You navigated the transition from password auth to Personal Access Tokens (PATs) and configured the Git credential manager (git config --global credential.helper store) within the Linux subsystem. This shows a deep understanding of terminal security protocols and environment configuration.
